Pointers in Rcpp

A hands-on walkthrough of pointers in C++ — what they are, why they matter, and how to use them safely inside Rcpp while calling your functions cleanly from R. Covers stack vs heap memory, when to use each, and the most common mistakes to avoid.

01 — Foundation

What is a pointer?

Every variable you declare in C++ lives somewhere in RAM. When you write int x = 42;, the compiler reserves a small slot in memory and puts the value 42 inside it. That slot has an address — a unique number that identifies exactly where in RAM it lives.

A pointer is a variable that stores that address. Instead of holding a value like 42, it holds a location like 0x61fe14. From that address you can read or modify whatever lives there — without ever using the original variable's name.

RAM ┌──────────────────────────┐ │ address │ value │ │ 0x61fe14 │ 42 ← x │ └──────────────────────────┘ ▲ p = 0x61fe14 (p stores the address) *p = 42 (dereferencing gives the value)
cppint x = 42;      // value stored on the stack
int* p = &x;     // p holds the ADDRESS of x
                 // &x means "give me the address of x"

*p = 100;        // dereference: write 100 to that address
// x is now 100 — same slot, new value

The & operator reads as "address of". The * operator reads as "value at".

02 — Memory

Stack memory vs heap memory

C++ gives you two places to store data: the stack and the heap. They behave very differently, and understanding the difference is essential before writing any serious Rcpp code.

Analogy The stack is like a stack of trays in a cafeteria. When a function starts, a new tray is placed on top. When the function ends, that tray is removed instantly — everything on it disappears automatically. The heap is like a huge warehouse. You walk in, grab a box (new), fill it with stuff, and when you're done you must return the box (delete). If you forget, the box sits there forever. That's a memory leak.
Property Stack Heap
AllocationAutomaticManual — new
CleanupAuto when scope exitsManual — delete
SpeedVery fastSlightly slower
RiskNoneLeaks if delete forgotten
Size limitSmall (a few MB)Large (available RAM)
 
cpp// Stack — automatic lifetime
int x = 42;          // created here
// ...               // lives until } closes
                     // freed automatically

// Heap — manual lifetime
int* h = new int(42);  // YOU allocate
*h = 100;              // use it
delete h;              // YOU must free it
h = nullptr;           // good practice: avoid dangling pointer

Rule for Rcpp: prefer stack variables for local temporaries. If you need heap memory, always pair every new with a delete — or better yet, use Rcpp::NumericVector which is heap-allocated but managed automatically by R's garbage collector.

03 — Why heap

Why you can't just use the stack for everything

The stack is fast and effortless, but it has three hard limits. Each one forces you to reach for the heap.

Problem 1 — The stack is small

The stack is typically only 1–8 MB total. That sounds like a lot until you're doing data analysis. One million doubles is already 8 MB.

size_problem.cpp ✗ crashes 
void bad_size() {
    double huge[5000000];   // ~40 MB — STACK OVERFLOW CRASH
}
size_fixed.cpp ✓ correct 
void good_size(int n) {
    double* huge = new double[n];   // heap: gigabytes available
    // ... use it ...
    delete[] huge;
    huge = nullptr;
}

// In Rcpp — even better, R manages the heap for you:
// [[Rcpp::export]]
NumericVector make_data(int n) {
    NumericVector out(n);   // heap, R GC frees it — no delete needed
    return out;
}

Problem 2 — Stack memory dies when the function ends

Stack memory belongs to its function's scope. The instant the function returns, that memory is gone. You cannot keep it alive or share it with the caller.

lifetime_problem.cpp ✗ dangling pointer 
double* bad_lifetime() {
    double x = 42.0;   // lives on the stack
    return &x;         // DANGER: x is destroyed after return!
}                      // caller now holds a pointer to dead memory
lifetime_fixed.cpp ✓ correct 
double* good_lifetime() {
    double* x = new double(42.0);   // heap — survives the function
    return x;                       // caller must delete when done
}

// In Rcpp — the natural way to return data to R:
// [[Rcpp::export]]
NumericVector make_vector(int n) {
    NumericVector result(n, 0.0);   // heap, R-managed
    return result;                  // R takes ownership
}

Problem 3 — The stack needs sizes known at compile time

The stack must know exactly how much space to reserve before your code even runs. In Rcpp, sizes always come from R at runtime — so you can never use a runtime variable as a stack array size.

size_unknown.cpp ✗ illegal in standard C++ 
void bad_unknown(int n) {
    double arr[n];   // ERROR: n is not known at compile time
}
size_unknown_fixed.cpp ✓ correct 
// std::vector — heap, auto-freed, exception-safe
void good_vec(int n) {
    std::vector<double> arr(n);
    arr[0] = 1.0;
}   // arr freed automatically here — even if an exception is thrown

// In Rcpp — the everyday pattern:
// [[Rcpp::export]]
NumericVector double_it(NumericVector x) {
    int n = x.size();          // stack: just a number
    NumericVector out(n);      // heap: size known only at runtime
    for (int i = 0; i < n; i++) {
        out[i] = x[i] * 2.0;
    }
    return out;
}
04 — Syntax

Understanding arr vs *arr

When you work with arrays and pointers, two expressions look similar but mean very different things:

int arr[] = {3, 7, 1, 9, 4}; index: 0 1 2 3 4 ┌────┬────┬────┬────┬────┐ │ 3 │ 7 │ 1 │ 9 │ 4 │ └────┴────┴────┴────┴────┘ addr: 0x100 0x104 0x108 0x10C 0x110 ↑ arr = 0x100 (the address of the first element) *arr = 3 (the value at that address)
cppRcout << arr;    // 0x100  — the address
Rcout << *arr;   // 3      — the value

arr++;           // move pointer to next address
Rcout << arr;    // 0x104
Rcout << *arr;   // 7

05 — Motivation

Why pointers exist — the problem they solve

In C++, function arguments are passed by value by default — the function receives a copy, and any changes it makes stay local. Pointers break that rule intentionally. 

cpp// Without pointer — broken
void broken(int a) {
    a = 100;    // changes local copy only
}               // caller's variable untouched

// With pointer — works
void works(int* a) {
    *a = 100;   // writes to the caller's actual address
}               // caller sees the change

This matters in three key situations:

  • Modifying multiple variables — return can only send back one value; pointers let you change as many variables as you need.
  • Working with arrays — arrays are passed as pointers automatically; you can walk through them without copying anything.
  • Sharing state across calls — a pointer passed to every recursive call ensures they all read and write to the same memory location.
06 — Stack in depth

Detailed stack examples

Before moving to heap-heavy patterns, it helps to see exactly what "stack only" looks like in real Rcpp code.

Example 1 — Pure scalar computation

When a function only needs simple numbers, everything stays on the stack. The compiler handles allocation and cleanup automatically — there is literally no allocation cost. 

cpp// [[Rcpp::export]]
double weighted_mean(NumericVector x, NumericVector w) {
    int    n   = x.size();         // stack: tiny int
    double num = 0.0, den = 0.0;   // stack: accumulators

    for (int i = 0; i < n; i++) {
        double xi = x[i], wi = w[i];   // stack: one value at a time
        num += xi * wi;
        den += wi;
    }

    double result = num / den;   // stack: final answer
    return result;
    // ALL stack variables freed here automatically
    // x and w (heap) are freed by R's GC
}

When to use stack-only: any function that takes data from R and returns a single scalar. Every intermediate value — counters, accumulators, loop variables — is a plain number that belongs on the stack. Maximum speed, zero allocation, zero cleanup code.

Example 2 — Stack struct for related temporaries

If you have a few related scalars that travel together through a calculation, a small struct keeps them tidy — and the whole struct still lives on the stack. 

cppstruct Stats {         // stack-allocated struct
    double mean;
    double variance;
    int    n;
};

Stats compute_stats(NumericVector x) {
    Stats s;              // entire struct on the stack
    s.n = x.size();
    s.mean = 0.0;
    for (int i = 0; i < s.n; i++) s.mean += x[i];
    s.mean /= s.n;

    s.variance = 0.0;
    for (int i = 0; i < s.n; i++)
        s.variance += pow(x[i] - s.mean, 2);
    s.variance /= s.n;

    return s;   // copied out, stack frame released automatically
}

Keep structs small — a few scalars at most. If a struct field is a NumericVector, that field's data lives on the heap even though the struct shell itself is on the stack.

07 — Practice

Four pointer challenges in Rcpp

The best way to understand pointers is to write them. Below are four classic exercises, each highlighting a different aspect of pointer usage.

Challenge 1 — Swap two numbers

Swapping two variables is impossible with a single return value. Pointers make it straightforward: pass the addresses of both variables and write through them directly.

swap_ptr.cpp ✓ correct 
#include <Rcpp.h>
using namespace Rcpp;

void swap_ptr(int* a, int* b) {
    int temp = *a;   // save value at a
    *a = *b;         // write b's value into a's address
    *b = temp;       // write saved value into b's address
}

// [[Rcpp::export]]
NumericVector swap_numbers(double a, double b) {
    int x = (int)a, y = (int)b;
    swap_ptr(&x, &y);          // pass addresses — works inside C++
    return NumericVector::create(x, y);
}
Before: a → │ 10 │ b → │ 20 │ temp = *a → temp = 10 *a = *b → a → │ 20 │ *b = temp → b → │ 10 │ ✓ swapped

Notice that void is used — no return value is needed because the function writes directly to the addresses it received. The changes are already in RAM when the function exits.

Challenge 2 — Sum of an array

Walking through an array with a pointer instead of an index is a fundamental C++ pattern. Each arr++ advances the pointer by exactly one int-sized step in memory.

sum_array.cpp ✓ correct 
int sum_ptr(int* arr, int size) {
    int sum = 0;
    for (int i = 0; i < size; i++) {
        sum += *arr;   // read value at current address
        arr++;         // advance to next address
    }
    return sum;
}

// [[Rcpp::export]]
int sum_array(NumericVector v) {
    int* arr = new int[v.size()];
    for (int i = 0; i < v.size(); i++) arr[i] = (int)v[i];
    int result = sum_ptr(arr, v.size());
    delete[] arr;
    return result;
}

Challenge 3 — Find the maximum value

The same walking pattern applies here. Start by assuming the first element is the maximum, then update whenever you find something larger.

find_max.cpp ✓ correct 
int max_ptr(int* arr, int size) {
    int max = *arr;                 // assume first element is max
    for (int i = 0; i < size; i++) {
        if (*arr > max) max = *arr; // update if larger found
        arr++;
    }
    return max;
}

// [[Rcpp::export]]
int find_max(NumericVector v) {
    int* arr = new int[v.size()];
    for (int i = 0; i < v.size(); i++) arr[i] = (int)v[i];
    int result = max_ptr(arr, v.size());
    delete[] arr;
    return result;
}

Challenge 4 — Reverse an array in place

This challenge introduces the two-pointer technique: one pointer starts at the beginning of the array, another at the end, and they walk toward each other swapping values until they meet in the middle. Notice that arr + size - 1 gives the address of the last element — size - 1 because arrays are zero-indexed.

reverse_array.cpp ✓ correct 
void reverse_ptr(int* arr, int size) {
    int* start = arr;              // address of first element
    int* end   = arr + size - 1;   // address of last element

    while (start < end) {
        swap_ptr(start, end);      // reuse our swap function!
        start++;                   // move inward from left
        end--;                     // move inward from right
    }
}

// [[Rcpp::export]]
NumericVector reverse_array(NumericVector v) {
    int size = v.size();
    int* arr = new int[size];
    for (int i = 0; i < size; i++) arr[i] = (int)v[i];
    reverse_ptr(arr, size);
    NumericVector result(size);
    for (int i = 0; i < size; i++) result[i] = arr[i];
    delete[] arr;
    return result;
}
arr = {1, 2, 3, 4, 5} size = 5 start↓ ↓end │ 1 │ 2 │ 3 │ 4 │ 5 │ swap → │ 5 │ 2 │ 3 │ 4 │ 1 │ start↓ ↓end │ 5 │ 2 │ 3 │ 4 │ 1 │ swap → │ 5 │ 4 │ 3 │ 2 │ 1 │ start↓↑end │ 5 │ 4 │ 3 │ 2 │ 1 │ start == end → stop ✓
08 — Rcpp patterns

Real Rcpp patterns — stack, heap, and pointers together

In production Rcpp code you rarely choose just one memory model — you use all three in the same function, each for the job it is best suited to.

Mental model Rcpp is like having a very smart assistant (R's GC) who manages the warehouse (heap) for you. You just say "I need a NumericVector of size n" and it handles the reservation, tracking, and cleanup. You focus on the actual computation, using the stack for all your arithmetic scaffolding.

Pattern 1 — Stack for scalars, GC heap for output

cpp// [[Rcpp::export]]
NumericVector running_sum(NumericVector x) {
    int n = x.size();          // stack: just an int
    NumericVector out(n);      // heap: R GC owns it
    double total = 0.0;        // stack: accumulator

    for (int i = 0; i < n; i++) {
        total  += x[i];
        out[i]  = total;
    }
    return out;   // R takes ownership — no delete needed ever
}

Pattern 2 — Raw pointer for fast inner-loop access

x.begin() returns a raw pointer to the first element of R's internal data. Using it directly in a tight loop avoids repeated bounds-checking and is the fastest way to process large vectors. 

cpp// [[Rcpp::export]]
NumericVector scale_fast(NumericVector x, double factor) {
    int n = x.size();
    NumericVector out(n);
    const double* xp = x.begin();    // raw pointer into R's heap data
    double*       op = out.begin();  // raw pointer into output

    for (int i = 0; i < n; i++) {
        op[i] = xp[i] * factor;      // direct memory access — fastest
    }
    return out;
}

Pattern 3 — std::vector for scratch buffers

When you need temporary heap memory that R should never see, prefer std::vector over raw new. Its destructor runs automatically on every exit path — including early returns and exceptions. 

cpp// [[Rcpp::export]]
NumericVector smooth(NumericVector x, int w) {
    int n = x.size();
    std::vector<double> buf(n);   // heap scratch buffer — auto-freed

    for (int i = 0; i < n; i++) {
        double sum = 0.0;
        int    cnt = 0;
        for (int j = i - w; j <= i + w; j++) {
            if (j >= 0 && j < n) { sum += x[j]; cnt++; }
        }
        buf[i] = sum / cnt;
    }

    return NumericVector(buf.begin(), buf.end());
    // buf freed here automatically — even if an exception was thrown above
}
09 — Integration

Calling Rcpp from R — and the pointer boundary problem

One of the first surprises when bringing pointer-based code into Rcpp is that raw pointers cannot cross the R/C++ boundary. If you mark a function with // [[Rcpp::export]] and give it an int* argument, you will get a compile error: 

errorerror: cannot convert 'SEXP' to 'int*' in initialization

This happens because R passes objects as SEXP — R's internal representation — and Rcpp does not know how to turn an SEXP into a raw int*. The solution is to keep pointer logic inside C++ and use Rcpp types at the boundary:

LocationUseWorks?
R → C++ argumentint*❌ compile error
R → C++ argumentint / NumericVector✓
Inside C++ onlyint*, &x✓ works freely
C++ → R returnint*❌ compile error
C++ → R returnNumericVector✓

The clean pattern is a two-layer design: an internal C++ function that uses pointers freely, and a thin exported wrapper that speaks Rcpp types: 

cpp// Layer 1 — internal, not exported, uses pointers freely
void swap_ptr(int* a, int* b) {
    int temp = *a; *a = *b; *b = temp;
}

// Layer 2 — exported, speaks R's language
// [[Rcpp::export]]
NumericVector swap_numbers(double a, double b) {
    int x = (int)a, y = (int)b;
    swap_ptr(&x, &y);       // & works fine INSIDE C++
    return NumericVector::create(x, y);
}

Also note that R is pass by value by default. Even if you could pass a pointer, R would copy the variable before sending it — so changes inside C++ would not affect R's copy. The correct idiom is always to return the modified values as a vector and reassign in R: 

rlibrary(Rcpp)
sourceCpp("pointer_practice.cpp")

# swap
result <- swap_numbers(10, 20)
x <- result[1]   # 20
y <- result[2]   # 10

# sum
sum_array(c(1, 2, 3, 4, 5))      # 15

# max
find_max(c(3, 7, 1, 9, 4))       # 9

# reverse
reverse_array(c(1, 2, 3, 4, 5))  # 5 4 3 2 1

# running sum
running_sum(c(1, 2, 3, 4, 5))    # 1 3 6 10 15
10 — Common mistakes

The four bugs that catch every beginner

Memory bugs are silent. The code compiles fine, often even runs fine most of the time — then crashes on a larger dataset. Knowing these patterns in advance is the only defence.

Mistake 1 — Returning a stack address

mistake1.cpp✗ undefined behaviour
double* bad() {
    double x = 42;
    return &x;   // x is gone — caller holds a pointer to dead memory
}
mistake1_fixed.cpp✓ correct
NumericVector good() {
    return NumericVector::create(42.0);   // heap, R-managed
}

Mistake 2 — Memory leak via early return

mistake2.cpp✗ leak on early return
void bad(int n) {
    double* buf = new double[n];
    if (n == 0) return;   // BUG: buf never deleted!
    delete[] buf;
}
mistake2_fixed.cpp✓ correct
void good(int n) {
    std::vector<double> buf(n);   // destructor runs on ANY exit path
    if (n == 0) return;           // safe — std::vector cleans up
}

Mistake 3 — delete vs delete[]

mistake3.cpp✗ wrong delete form
double* arr = new double[100];
delete arr;     // WRONG — must use delete[] for arrays
mistake3_fixed.cpp✓ correct
double* arr = new double[100];
delete[] arr;   // correct — matches new[]
arr = nullptr;

Mistake 4 — Using a pointer after delete

mistake4.cpp✗ dangling pointer
int* p = new int(5);
delete p;
*p = 10;        // undefined behaviour — p points to freed memory
mistake4_fixed.cpp✓ correct
int* p = new int(5);
delete p;
p = nullptr;    // always set to nullptr after delete
// *p = 10;    // would now crash cleanly instead of silently corrupting

The rule that prevents 90% of bugs: avoid raw new/delete entirely. Use NumericVector for R data, std::vector for scratch buffers. Let the destructors do the work.

11 — Decision guide

When to use which — a quick reference

Situation Use Why
Simple number inside a function int x = 5; Stack — zero cost, auto-freed
Data returned to R NumericVector GC heap — R manages lifetime
Data passed in from R NumericVector R already owns the memory
Scratch buffer, unknown size std::vector<double> Heap, auto-freed on any exit
Fast inner-loop access to vector x.begin() pointer Direct memory access, fastest
C library needs raw pointer new / delete Only when forced — pair carefully

The one-sentence version: stack for numbers you calculate with, NumericVector for data that touches R, std::vector for scratch buffers — and never write raw new if you can avoid it.

12 — Summary

Key rules to remember

  • arr is the address — where something lives. *arr is the value — what lives there.
  • Assigning x = 100 or *p = 100 both change the same memory cell. No copy is made.
  • Use void as the return type when a function modifies variables through pointers — nothing needs to come back because the work is done directly in RAM.
  • Raw pointers (int*) cannot appear in // [[Rcpp::export]] function signatures. Keep pointer logic in unexported internal functions.
  • Always delete[] anything you allocated with new[]. For most Rcpp work, prefer NumericVector and std::vector and avoid manual heap management altogether.
  • arr + size - 1 points to the last element because arrays are zero-indexed. arr + size would be one past the end — undefined behaviour.
  • After any delete, immediately set the pointer to nullptr to prevent accidental use of freed memory.
  • Prefer std::vector over raw new[] for scratch buffers — its destructor runs on every exit path, including exceptions.

The patterns covered here — swap, sum, max, reverse, running sum, smooth — represent the core techniques that appear repeatedly in real Rcpp work: writing through pointers, walking arrays, the two-pointer technique, scratch buffers, and bridging cleanly between R and C++. Mastering these makes the step to more advanced topics like custom data structures and memory-efficient algorithms much more natural.
